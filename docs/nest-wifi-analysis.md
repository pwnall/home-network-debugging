# Nest WiFi Pro Ethernet Backhaul Analysis

This document outlines the packets and protocols identified during network monitoring of a Nest WiFi Pro unit (WiFi 6E). These packets are likely used to probe, discover, and maintain Ethernet backhaul connections between mesh units.

## Identified Probing Mechanisms

### 1. Spanning Tree Protocol (STP)
The device consistently broadcasts STP (802.1d or 802.1w) BPDUs.
*   **Frequency:** Every 2 seconds.
*   **Destination MAC:** `01:80:c2:00:00:00` (STP Multicast).
*   **Source MAC Prefix:** `b8:7b:d4` (Google).
*   **Root Identification:** The primary router typically elects itself as the **Root Bridge** with a priority of **30976** (hex `0x7900`). Other units on the segment will forward BPDUs identifying this Root ID.
*   **Function:** Likely used for loop detection and to determine if a wired path exists between two mesh units. The presence of these packets on the wire is a strong indicator of a direct Ethernet link.

### 2. Specialized DHCP Probes
The Nest device sends DHCP Discover packets with a unique vendor-specific string.
*   **Packet Type:** DHCP Discover (Port 67/68).
*   **Sname Field:** `gwifi_rouge_dhcp_detection`
*   **Function:** While named "rouge" (likely "rogue") detection, this specific string identifies the sender as a Google Nest/WiFi device. It is likely used to map the network topology and detect other Google-managed nodes on the segment.

### 3. Multicast DNS (mDNS)
Extensive mDNS traffic is observed on port 5353.
*   **Service Types:** 
    *   `_googlecast._tcp.local`
    *   `_matter._tcp.local`
    *   `_ipp._tcp.local`
*   **Function:** Discovery of peer devices. Nest units use mDNS to advertise their presence and capabilities to other units on the same layer-2 segment.

### 4. IPv6 Neighbor Discovery & Router Advertisements
Multiple components of the Nest device send ICMPv6 Router Advertisements.
*   **Destination:** `ff02::1` (All nodes).
*   **Source:** Link-local `fe80::` addresses derived from the Google MAC.
*   **Function:** Establishing the IPv6 "fabric" connectivity that Google uses for internal communication between mesh nodes.

### 5. Simple Service Discovery Protocol (SSDP)
When acting as a router, the Nest device broadcasts SSDP messages.
*   **Packet Type:** UDP (Port 1900).
*   **Destination:** `239.255.255.250`.
*   **Function:** Used for Universal Plug and Play (UPnP) discovery, advertising itself as an Internet Gateway Device (IGD) to the local network.

### 6. Address Resolution Protocol (ARP)
The Nest device frequently queries the network for other IP addresses using ARP.
*   **Packet Type:** ARP Request.
*   **Function:** Resolving IP addresses of local clients and detecting connected devices on the LAN segment.

### 7. Absent Protocols (What is *not* used)
During an exhaustive analysis of the traffic generated by Nest WiFi Pro devices across the network, several standard topology and mesh protocols were confirmed to be **absent**:
*   **LLDP (Link Layer Discovery Protocol):** Not emitted by Nest units.
*   **IEEE 1905.1 (EasyMesh):** Not utilized for mesh discovery or backhaul setup.
*   **IGMP (Internet Group Management Protocol):** Nest units do not use standard IGMP joins for their multicast discovery traffic.
Because these standard Layer 2/3 topology protocols are missing, **STP (and specifically the Root Path Cost)** is the sole determinative mechanism for Ethernet backhaul election.

### 8. Application-Layer Broadcasts (Informational)
While analyzing traffic, two other broadcast types from Google MACs were frequently observed. These are application-layer protocols and do not affect backhaul topologies:
*   **TP-Link Smart Home Discovery:** UDP broadcasts to Port 9999 containing JSON (`{"system":{"get_sysinfo":{}}}`). This is Google Assistant proactively scanning for local smart plugs.
*   **Google NPE Audio Sync:** UDP broadcasts to Ports 9478/9487 containing hex-encoded ASCII (`GoogleNPE_...`). This is used for multi-room Cast audio synchronization.

## Detection Guide for AI Agents

To detect Nest WiFi Pro backhaul probing, the recommended method is to use the `nest-check` tool provided in this repository.

### Specialized Tool: `nest-check`
A Rust-based utility (`crates/nest-wifi-pro-ethernet`) is available to identify these packets and extract relevant metadata.

**Execution:**
```bash
# Recommended duration is at least 60 seconds to catch infrequent probes
./target/debug/nest-check --interface [interface] --duration 60
```

## Detection via an Ethernet Switch

When the Nest WiFi Pro and the monitoring host are connected through an Ethernet switch (rather than a direct cable), the following behaviors must be considered:

### 1. Multicast and Broadcast Propagation
*   **STP**: Managed switches participating in RSTP will consume the Nest Router's BPDUs and emit their own. Verification tools should look for the **Root ID** inside the BPDUs rather than just the source MAC of the packet to identify the Nest Router.
*   **STP Priority (The Split-Brain Problem)**: A critical issue occurs if the switch priority is set too high (e.g., `61440`). The Nest Router dynamically changes its STP priority between `30976` and `31488`. Meanwhile, secondary points might maintain a priority of `31232`. If the switch has a priority of `61440`, it will elect the point with `31232` as the Root Bridge over Ethernet. However, over the Wi-Fi mesh, the Router might advertise itself as the Root. This creates a split-brain topology where the Ethernet Root is different from the Wi-Fi Root, causing points to block their Ethernet ports and fall back to "Wireless" backhaul.
    *   **Resolution**: To permanently fix this and force all Nest units to use wired backhaul, **the Ethernet Switch must be configured as the Global Root Bridge**. Set the switch's STP Priority to `4096`. This extremely low priority guarantees the switch wins the root election against all Nest units. As a result, all Nest units will see the switch as the Root Bridge and select their Ethernet ports as their Root Ports, guaranteeing wired backhaul.
*   **Loop Prevention, Edge Ports, & Path Cost**: Because Nest units maintain a wireless mesh, managed switches will detect a loop between the wired and wireless paths. Secondary Nest units advertise a superior Root Path Cost via their wireless connection compared to the default wired path cost provided by a switch (e.g., 20000 for a 1G link). This causes the switch to yield and place ports connected to secondary units in an `Alternate/Discarding` state. 
    *   **Resolution**: First, configure all ports intended for secondary Nest units as **Edge Ports** (Admin Edge). Second, and most critically, artificially lower the switch's perceived distance to the Root Bridge by setting the **Path Cost** of the switch's Root Port (the uplink to the primary Nest router) to `1`. This near-zero path cost forces the switch to become the Designated bridge for all secondary links, causing the secondary Nest units to prefer the wired connection and block their wireless backhaul.
*   **Network Restart Requirement**: Nest WiFi units cache their topological state (like the known Root Path Cost and established wireless backhaul links). If the switch is reconfigured while the units are running, they will often fail to dynamically renegotiate their backhaul to the new, superior wired path and remain stuck in a "Wireless" state.
    *   **Resolution**: **Always restart the entire Nest WiFi network** (or reboot all units simultaneously) *after* the switch's STP Priority and Path Cost settings have been correctly applied and saved. This forces the mesh to build its topology from scratch and discover the superior wired switch paths during its initial boot sequence.
*   **mDNS and SSDP (Multicast)**: With IGMP Snooping disabled, the switch successfully flooded multicast traffic (`_matter`, `_googlecast` on port 5353, and SSDP on port 1900) to all ports, including the monitoring host.
*   **DHCP/RAs/ARP (Broadcast & IPv6 Multicast)**: These packets (including the `gwifi_rouge_dhcp_detection` probe and standard ARP requests) were observed passing through the switch as expected since they are flooded to the broadcast domain.

### 2. Detection Strategies for Switched Environments
*   **RSTP Port Roles**: The `rstp-ports` command on the TRENDnet switch is a highly reliable indicator. If the port connected to the Nest unit is in the **Root** role, it confirms the switch has accepted the Nest unit as the Spanning Tree Root.
*   **Tooling**: The `nest-check` tool correctly identifies the relationship between the Switch (acting as a bridge) and the Nest Router (acting as the Root).

### 3. Switch-Induced Artifacts
*   **VLAN Tags:** Some switches might strip or add VLAN tags (802.1Q). Detection logic should be robust enough to handle both tagged and untagged frames (`tcpdump -e` helps identify if tags are present).
*   **Latency/Jitter:** Switches introduce minor timing variations. Avoid relying on precise microsecond inter-packet timing for fingerprinting if a switch is in the path.

## Analysis Methodology
Findings were derived by monitoring the `eth-main` interface (`enx9cbf0d005f94`) on the primary host while connected directly to the Nest WiFi Pro. Traffic was captured using `tcpdump` and analyzed for non-standard or Google-specific signatures.

*   **First Pass (Mesh Node):** Analyzed the Nest WiFi Pro acting as a secondary mesh node. Raw captures are available in `data/nest_backhaul.pcap` and `data/nest_backhaul_large.pcap`.
*   **Second Pass (Router Mode):** Analyzed the Nest WiFi Pro acting as the primary router, directly connected to a cable modem. A new raw capture was collected, available in `data/router_mode.pcap`, and analyzed to identify additional packets like SSDP and ARP.

### Packet Emission Periodicity
Analysis of the `router_mode.pcap` timestamps revealed the following intervals for critical discovery and keepalive packets:
*   **STP (Spanning Tree Protocol):** Every 2 seconds.
*   **ARP (Address Resolution Protocol):** Frequent bursts (every 1-10 seconds) as the router queries the LAN.
*   **DHCP Rouge Probes:** Exactly every 30 seconds.
*   **SSDP (Simple Service Discovery Protocol):** Exactly every 60 seconds.

Because of the 60-second SSDP cycle, any diagnostic capture or tool execution must run for at least 65 seconds to guarantee observing all packet types emitted by the Nest WiFi Pro.
